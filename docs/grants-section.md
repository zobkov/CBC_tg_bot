## General
Dialog path: `app/bot/dialogs/grants`

## States

`GrantsSG`:
- `MAIN_GENERAL` — general main window
- `MAIN_GSOM`  — menu for approved users
- `GRANT_INFO` — grant competition info 
- `COURSE` — educational course info
- `FAQ` — grant-specific FAQ 
- `GRANT_INFO_GSOM` — grant competition info for gsom approved
- `COURSE_GSOM` — educational course info for gsom approved
- `FAQ_GSOM` — grant-specific FAQ for gsom approved
- `MENTOR`
- `LESSON`

## Implementation details
Considering GSOM dialog branch:
1. New DB table – `user_mentors`
   1. `user_id` – FK to users (on delete cascade)
   2. `lessons_approved` (jsonb list of all approved tags)
   3. `mentor_contacts`
2. On entering this dialog a handler should process user (as of now main entrypoint – guest main). If they are in a user_mentors table - start with state `MAIN_GSOM`. Otherwise start with `MAIN`. 
3. Lesson config should be stored in `grant_lessons.json` config. Should have:
   1. `tag`
   2. `name`
   3. `description`
4. Lessons list (`LESSON` state) is generated from `grant_lessons.json`. Format: `{approval_status} {name}`
5. Have to read `grant_lessons.json` on bot startup. Then access it in dialog_data. On fail initiate reread. Also need command: `/config_reset` accessible in admin_lock router. Forcefully rereads lessons' config and rewrites in dialog_data to update or fix. 
6. Need to create command in admin_lock router. `/grant {user_id} {lesson_tag}` – adds to list of that user new tag
